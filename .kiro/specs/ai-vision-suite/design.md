# AI Vision Suite — Design

## Architecture

The AI vision suite follows localpress's standard three-layer architecture
(CLI command layer → engine layer → adapter layer), plus the shared
history/state layer that every mutating command hooks into.

```text
┌─────────────────────────────────────────────────────────────────┐
│ CLI command layer  (src/cli/commands/*.ts)                       │
│   caption.ts  title.ts  describe.ts  classify.ts  tag.ts vision.ts│
│   - flag parsing, ID/bulk-selection resolution, dry-run posture   │
│   - per-command prompt/output formatting, --json shaping          │
└───────────────┬─────────────────────────────────────────────────┘
                │ calls
┌───────────────▼─────────────────────────────────────────────────┐
│ Engine layer  (src/engine/caption/*.ts)                          │
│   ollama.ts    — Ollama HTTP client, prompt building, response    │
│                  cleanup, garbage-output fallback                 │
│   run-bulk.ts  — shared per-item loop (caption/title/describe):   │
│                  fetch → skip-check → generate → snapshot → write │
│   types.ts     — VisionKind, CaptionOptions/Result shared types   │
└───────────────┬─────────────────────────────────────────────────┘
                │ calls (WpBackend interface)
┌───────────────▼─────────────────────────────────────────────────┐
│ Adapter layer  (src/adapters/{resolver,rest,wp-cli}.ts)          │
│   AdapterResolver.resolve('list' | 'get' | 'update-meta')         │
│   RestAdapter — always available (WP REST + Application Passwords)│
│   WpCliAdapter — opt-in, same capability surface for this suite   │
└─────────────────────────────────────────────────────────────────┘
```

Ollama itself is an external local HTTP service (`http://localhost:11434`
by default), not a first-party subsystem — `src/engine/caption/ollama.ts`
is a thin client against its `/api/generate` and `/api/tags` endpoints. No
adapter or engine code in this suite ever calls a cloud vision/LLM API.

The suite also touches two more localpress subsystems, invoked directly
from the command layer rather than through a dedicated adapter:
- **SQLite state** (`src/engine/state/db.ts`, `SiteDb`) — every processed
  attachment is upserted into the `attachments` table (keeping the FK for
  `processing_history` rows satisfied even on failure paths), and every
  attempt (success or failure) is recorded as a `processing_history` row.
  `classify` additionally uses `processing_history.paramsJson` as an ad hoc
  cache for the last classification result, read back later by `optimize`
  via `getCachedClassification()` in `classify.ts`.
- **Time-machine / history** (`src/engine/history/index.ts`) — `caption`,
  `title`, `describe`, `tag`, and `vision` (when `--apply`d) open a history
  session before their mutating loop and capture a metadata-only snapshot
  (previous `altText`/`title`/`caption`/`description`) per item before each
  write, so `localpress undo` can restore the prior values.

## Key files and modules

| Path | Responsibility |
|---|---|
| `src/cli/commands/caption.ts` | `caption` command: alt-text generation. Owns its own inline bulk loop (not `run-bulk.ts`) — it predates the shared runner and adds `--fallback-model`, `--list-models`, and richer per-item logging. |
| `src/cli/commands/title.ts` | `title` command: thin wrapper around `runBulkVision()` with `kind: 'title'`, `buildUpdate: title`, and the `looksAutoGenerated()` heuristic for `--missing-title`. |
| `src/cli/commands/describe.ts` | `describe` command: thin wrapper around `runBulkVision()` with `kind: 'description'`. |
| `src/cli/commands/classify.ts` | `classify` command: single-purpose classifier, no bulk mode, writes nothing to WP by default. Also exports `getCachedClassification()`, consumed by `optimize`. |
| `src/cli/commands/tag.ts` | `tag` command: generates and writes a `[tags: …]` block into the WP `caption` field. Has its own inline loop (predates `run-bulk.ts`, similar shape to `caption.ts`). |
| `src/cli/commands/vision.ts` | `vision` command: composed single-pass generator across `alt`/`title`/`description`/`tags`/`classify`; print-only unless `--apply`. Its own inline loop, calling `generateCaption()` once per requested field per item. |
| `src/engine/caption/ollama.ts` | Ollama HTTP client (`generateCaption`, `generateCaptionWithFallback`), per-kind prompt templates (`buildPrompt`), response cleanup (`cleanResponse`/`cleanCaption`/`cleanTitle`/`cleanClassify`/`cleanTags`/`cleanTagsArray`), the garbage-output heuristic (`looksLikeGarbage`), and pre-send image downscaling/format conversion (`downscaleForVision`). |
| `src/engine/caption/run-bulk.ts` | `preflightOllama()` (availability + installed-model check, shared by every command in this suite) and `runBulkVision()` (shared per-item loop used by `title.ts` and `describe.ts`). |
| `src/engine/caption/types.ts` | `VisionKind` (`alt`/`title`/`description`/`classify`/`tags`), `CaptionOptions`, `CaptionResult`. |
| `src/cli/mcp/tools.ts` | MCP tool definitions that shell out to the same CLI commands: `caption`, `generate_title` (→ `title`), `generate_description` (→ `describe`), `vision`, `tag`, `classify`. Each tool builds an argv array and re-invokes the CLI in-process (`runCli`/`runCliBatched`), so the MCP surface has no separate business logic — it is a typed front door onto the same command implementations. |

## Data flow

### Single-item flow (e.g. `localpress caption 123`)

1. **CLI layer**: `caption.ts` resolves the active site/config, builds an
   `AdapterResolver`, and resolves three capabilities: `list`, `get`,
   `update-meta`.
2. Resolves the effective model (`--model` > `config.defaults.captionModel`
   > `moondream`) and calls `isOllamaAvailable()` then `listOllamaModels()`
   as a pre-flight check (fails fast with remediation text if Ollama is
   down or the model isn't pulled).
3. For attachment `123`: `getAdapter.getMedia(123)` fetches WP metadata
   (URL, existing `alt_text`, MIME type, etc.) over REST or WP-CLI.
4. The attachment row is upserted into SQLite (`db.upsertAttachment`) —
   done unconditionally, before any network/Ollama work, so a later
   failure can still record a `processing_history` row without hitting a
   foreign-key violation.
5. If `alt_text` is already set and `--overwrite` wasn't passed, the item
   is skipped here (no Ollama call, no write).
6. The image bytes are downloaded (`fetch(item.url)`), and
   `generateCaptionWithFallback()` is called:
   - `downscaleForVision()` resizes the buffer to ≤1024px on its long edge
     and re-encodes to PNG if the source isn't PNG/JPEG (via lazy-loaded
     `sharp`).
   - The buffer is base64-encoded and POSTed to
     `${ollamaUrl}/api/generate` with the per-kind prompt, `stream: false`,
     `num_predict: 200`, and a 120s abort timeout.
   - The raw `response` string is cleaned via `cleanResponse()` →
     `cleanCaption()` for the `alt` kind.
   - If `--fallback-model` is configured and `looksLikeGarbage()` flags the
     result, the same image is retried against the fallback model.
7. If not a dry run: a metadata-only time-machine snapshot of the
   *previous* alt/title/caption/description is captured, then
   `updateAdapter.updateMetadata(123, { altText: result.caption })` writes
   the new alt text to WordPress (REST `POST /wp/v2/media/123` under
   `RestAdapter`, or the WP-CLI equivalent under `WpCliAdapter`).
8. A `processing_history` row is recorded (`operation: 'caption'`, model,
   duration, success/failure) regardless of dry-run state (dry runs record
   nothing here since the whole conditional block is skipped along with
   the write — see Edge cases below).
9. Results are accumulated and printed as a human summary or, with
   `--json`, as a structured object (`{ dryRun, processed, skipped,
   failures, results }`).

### Bulk flow (`--missing-alt`, `--all`, or `title`/`describe` bulk modes)

Same per-item steps, but the ID list is first built by paging through
`listAdapter.listMediaPage({ type: 'image/', perPage: 100, page })` until
`page >= totalPages`, then filtered client-side (missing alt / auto-
generated title / missing description / missing tag block). The run
defaults to dry-run (`resolveDryRun(parentOpts, isBulk)` — see below) unless
`--apply` is passed; `--dry-run` can also force preview mode on an
otherwise-immediate explicit-ID run.

### `vision` flow (composed, per-item)

For each ID, `vision.ts` downloads the image once and then calls
`generateCaption()` once per requested field (`--fields`, default all
five), building up an in-memory proposal object. In print-only mode (the
default) nothing further happens. With `--apply`, a single `UpdateMetadata`
object is assembled from only the fields that are unset (or all fields, if
`--overwrite`) and written in one `updateMetadata()` call per attachment —
so `vision --apply` never issues more than one WordPress write per item
even though it may have made up to five Ollama calls.

## Key design decisions

- **Local Ollama over a cloud vision API** (`CLAUDE.md` → "AI captioning /
  vision" in "Locked architectural decisions"). Every command in this
  suite talks HTTP to a user-controlled Ollama instance, never to a
  managed cloud API. This is deliberate: no per-image cost, no data
  leaving the user's machine, and the user owns model choice/quality
  tradeoffs. The tradeoff, visible directly in the code, is that output
  quality is uneven across models — hence the extensive post-processing
  (`cleanCaption`/`cleanTitle`/`cleanClassify`/`cleanTags`) and the
  garbage-output fallback mechanism (Requirement 11) exist specifically to
  paper over small local models (moondream et al.) producing inconsistent
  output compared to what a hosted frontier vision API would return more
  reliably. `moondream` (~1.7GB) is the recommended default for speed;
  `llava`/`llava-llama3` is offered as a higher-quality, slower option —
  users trade latency for quality locally rather than trading money for
  quality via an API tier.
- **Two implementations of the same bulk loop.** `title.ts` and
  `describe.ts` share `runBulkVision()` in `run-bulk.ts`. `caption.ts` and
  `tag.ts` implement their own inline loops with the same shape
  (upsert-first, skip-if-set, fetch, generate, snapshot, write, record).
  This is not a design decision so much as an observed inconsistency in
  the shipped code — `caption.ts`'s header comment even notes `run-bulk.ts`
  exists, but `caption` predates it and has extra behavior (`--fallback-model`,
  `--list-models`) that wasn't folded back into the shared runner. Treat
  `run-bulk.ts` as the current template for new fields, but don't assume
  `caption.ts`/`tag.ts` internals match it exactly.
- **Time-machine integration is metadata-only for this suite.** Unlike
  `optimize`/`remove-bg` (which snapshot binary blobs of the original file),
  every command here snapshots only the four text metadata fields
  (`altText`, `title`, `caption`, `description`) before mutation — cheap
  enough that history is effectively free for these commands. Snapshots
  and the enclosing session are skipped entirely when the run is a dry run
  (no writes means nothing to make undoable).
- **`classify` and `tag` write to fields that don't require WP taxonomy
  registration.** `classify` caches its result only in local SQLite
  (`processing_history.paramsJson`), never in WordPress, specifically so
  `optimize` can consume it without any WP-side schema requirement.
  `tag` writes into the `caption` field (present on any `attachment` post
  by default) using an inline `[tags: …]` marker instead of a real
  taxonomy term, because attachment taxonomies aren't registered on a
  stock WordPress install — the code comment in `tag.ts` explicitly frames
  real-taxonomy support as a possible future iteration contingent on
  `doctor --plugins` detection.
- **Bulk safety follows the shared `resolveDryRun` convention** (`CLAUDE.md`
  → "Bulk safety"). `caption`, `title`, `describe`, `tag`, and `vision` all
  default explicit-ID invocations to executing immediately, and default
  `--all`/`--missing-*` bulk invocations to dry-run unless `--apply` is
  passed. `vision` is a partial exception: even for explicit IDs it
  defaults to *print-only* (no `resolveDryRun` call at all — it checks
  `parentOpts.apply` directly) rather than "execute immediately," because
  its whole point is proposing a wall of AI-generated fields for review
  before any of them touch WordPress. `classify` has no dry-run concept at
  all since it never writes to WordPress.
- **Pre-flight model check before the bulk loop.** Both the inline
  (`caption.ts`) and shared (`preflightOllama()` in `run-bulk.ts`)
  implementations check not just that Ollama is running but that the
  *specific requested model* is installed, before processing the first
  attachment. This exists purely to avoid the failure mode of running a
  300-item bulk job that fails identically 300 times because of a typo'd
  or unpulled model name.

## Error handling / edge cases

- **Ollama unreachable**: fails before any per-item work, exit code 2,
  with `ollama serve` / `ollama pull <model>` remediation text and a
  setup-guide URL.
- **Model not installed**: pre-flight fails with a list of vision models
  that ARE installed (if any) plus the exact `ollama pull`/`config
  set defaults.captionModel` commands to fix it. If the pre-flight probe
  itself errors (not the check failing, but the check *throwing*), the run
  proceeds anyway and lets per-item failures surface normally — this is a
  deliberate "don't block on a flaky pre-check" choice.
- **Non-image MIME type**: skipped with a warning, not treated as a
  failure.
- **Image format incompatible with Ollama** (WebP/AVIF/GIF): converted to
  PNG via `sharp` before the request; if `sharp` itself is unavailable or
  throws, the original bytes are sent as-is (best-effort degrade, not a
  hard failure).
- **Oversized images**: downscaled to ≤1024px long edge before sending,
  both for speed and because small models can return empty responses on
  very large inputs (the moondream 4K-screenshot failure mode is called
  out directly in the `ollama.ts` comments).
- **Garbage/empty output**: an empty Ollama response is a hard error
  ("try a different --model"); a short/numeric/coordinate-looking response
  is caught by `looksLikeGarbage()` and triggers a fallback-model retry
  only if one is configured (Requirement 11) — otherwise it's accepted
  as-is.
- **Wedged/slow Ollama**: a 120-second `AbortSignal.timeout` on the
  `/api/generate` call raises a clear timeout error instead of hanging a
  bulk run indefinitely.
- **Per-item failure inside a bulk run**: caught, logged, recorded (best-
  effort) as a `processing_history` failure row, and the loop continues to
  the next ID; the command exits non-zero at the end if `failures > 0`, but
  every ID is still attempted.
- **FK-safety on the failure path**: `db.upsertAttachment()` runs
  unconditionally and *before* any network/Ollama call, specifically so
  that a later `db.recordProcessing()` failure-row insert (which has a
  foreign key on `attachments`) doesn't itself throw. The failure-row
  insert is additionally wrapped in its own try/catch as a last-resort
  guard ("we'd rather lose the failure breadcrumb than abort the bulk
  run").
- **`--fields` validation in `vision`**: an unrecognized field name or an
  empty resolved field list is rejected before any Ollama/WordPress call,
  exit code 2.

## Testing approach

- `test/unit/caption-clean.test.ts` — unit tests for `cleanCaption()` (quote
  stripping, meta-phrase intro stripping, multi-paragraph/bullet trimming,
  word-boundary truncation, a real llava/llama3.2-vision verbose-output
  regression case) and `cleanClassify()` (plain label match, negation
  avoidance — "a photograph, not a screenshot" — earliest-match-wins, and
  fallback-to-first-word).
- `test/unit/dry-run-honesty.test.ts` — static source-analysis test
  asserting that `caption.ts`, `title.ts`, `describe.ts`, `tag.ts`, and
  `vision.ts` each contain a recognized dry-run-gating pattern
  (`resolveDryRun`/`isDryRun`/`dryRun`/`parentOpts.apply`/`options.apply`).
  `classify.ts` is intentionally excluded from this list (it never
  mutates WordPress).
- `test/unit/dry-run-wiring.test.ts` — builds the full commander program
  (including `registerCaptionCommand`, `registerClassifyCommand`,
  `registerDescribeCommand`, `registerTagCommand`, `registerTitleCommand`,
  `registerVisionCommand`) and asserts no command in the tree redeclares
  `--dry-run`/`--apply` locally (they must come only from the shared root
  options).
- `test/unit/mcp.test.ts` and `test/unit/mcp-schema-cli-parity.test.ts` —
  assert the MCP tool registry includes `caption` and cross-check tool
  input schemas against the underlying CLI flags for parity.
- No dedicated integration test in `test/integration/` exercises
  caption/title/describe/classify/tag/vision against live Ollama — the
  Docker Compose integration harness (`test/integration/docker-compose.yml`)
  brings up WordPress + MySQL only, not an Ollama instance, and a
  targeted search of `test/integration/` for these command names turned up
  no matches at the time this doc was written. This means the
  WordPress-write path for this suite (`updateMetadata` calls,
  time-machine snapshot/restore round-trips) is covered only indirectly —
  via the shared `RestAdapter`/`WpCliAdapter` integration tests for
  `updateMetadata` in general, not through the vision commands themselves.
  Flagging this as a real gap rather than asserting coverage that doesn't
  exist.
