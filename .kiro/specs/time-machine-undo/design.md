# Time-Machine / Undo — Design

## Architecture

Time-machine/undo is not a standalone feature with its own top-level workflow — it is a cross-cutting engine module that every destructive CLI command (and the equivalent MCP tools) calls into, following localpress's standard layering:

```
CLI command layer        src/cli/commands/{optimize,convert,resize,remove-bg,caption,...}.ts
                          src/cli/commands/history.ts, undo.ts
                          src/cli/components/HistoryBrowser.tsx (Ink TUI)
                                │
                                ▼
Engine layer              src/engine/history/{index,store,types}.ts  ← this subsystem
                                │  (SnapshotStore reads/writes SQLite directly;
                                │   no adapter involvement for read/write of history itself)
                                ▼
Adapter layer              src/adapters/{rest,wp-cli}.ts via AdapterResolver
                          (only touched during RESTORE — replaceInPlace / updateMetadata / upload)
                                │
                                ▼
State layer                src/engine/state/{schema,db}.ts (SiteDb, bun:sqlite)
Blob storage                <config-dir>/snapshots/<site>/<session-id>/<attachmentId>-<rowId><ext>
```

Two distinct data paths exist:
- **Capture path** (used by every mutating command): command → `engine/history` helpers → `SnapshotStore` → SQLite (`sessions`/`snapshots` tables) + blob files. No adapter calls — capture happens purely against local storage, before the command talks to WordPress.
- **Restore path** (used only by `undo`): `undo` command → `SnapshotStore` (read snapshot + blob) → `AdapterResolver.resolve()/tryResolve()` → `RestAdapter`/`WpCliAdapter` → live WordPress mutation (`replaceInPlace`, `updateMetadata`, or `upload` as fallback).

The MCP server (`src/cli/mcp/tools.ts`) does not call the engine directly; its `history_list`, `history_show`, `undo`, and `history_prune` tools shell out to the CLI (`runCli(...)`) the same way most other MCP tools do, keeping one code path and one JSON contract. A `localpress://history` MCP resource (`src/cli/mcp/resources.ts`) exposes the same data read-only via `readAsResource(uri, ['history'])`.

## Data flow

**Capture (every mutating command, e.g. `optimize.ts`):**
1. Command resolves the active site and opens `SiteDb` + `openSnapshotStore(db, getConfigDir())`.
2. If `resolveHistoryConfig(config.history).enabled`, the command calls `openHistorySession(store, site.name, '<command>', <params>)` once, before the per-item loop — this inserts one `sessions` row and returns a `SessionRecord`.
3. For each item, the command performs its normal idempotency check first (e.g. `shouldSkipOptimize`). Only if the item will actually be mutated does it call `captureSnapshot(store, { siteName, sessionId, attachmentId, operation, sourceBytes, beforeHash, beforeMeta })` — this happens **after** the skip check and **before** the WordPress write, so skipped items never get a wasted snapshot and captured snapshots always precede the mutation they protect against.
   - `sourceBytes` non-null → `'binary'` snapshot: bytes are written synchronously to `<blobRoot>/<site>/<session>/<attachmentId>-<rowId><ext>` before `capture()` returns (durability — no async gap where the DB row could point at a not-yet-written file).
   - `sourceBytes` null → `'metadata-only'` snapshot: only the `before_meta` JSON column is populated; `blob_path`/`blob_size` stay null/0.
   - In-session dedupe: if `beforeHash` matches an existing un-restored binary snapshot for the same attachment in the same session, the existing snapshot ID is returned and no new row/blob is written.
4. After the loop, the command calls `closeHistorySession(store, session, { maxSizeBytes })`, which sets `finished_at`/`item_count` (or deletes the session row if `item_count` is 0) and then runs `store.prune(siteName, { maxSizeBytes })` to enforce the size cap.

**Restore (`undo` command):**
1. `undo` resolves target snapshot(s): a single ID (`--snapshot`), the latest un-restored snapshot for an attachment (`--attachment`), an explicit session prefix, or (default) `store.getLastSession(site.name)` — the newest session that still has at least one un-restored snapshot.
2. Session-targeted restores are bulk and dry-run by default (must pass `--apply`); `--snapshot`/`--attachment` targets execute immediately, matching the CLI-wide convention that explicit IDs bypass the dry-run gate.
3. For each targeted snapshot, `restoreSnapshot(snap, resolver, store, strict)` runs:
   - `'metadata-only'`: resolves an `'update-meta'`-capable adapter and calls `updateMetadata` with the recorded before-values.
   - `'binary'`: `store.readBlob(snap)` loads and integrity-checks the blob (existence, size match, SHA-256 match against `beforeHash` if recorded), throwing rather than returning suspect bytes. Then it tries `resolver.tryResolve('replace-in-place')`; if available, it fetches the attachment's *current* metadata (`resolve('get').getMedia`) to detect a format change since the snapshot (e.g. optimize converted PNG→WebP) and passes `newExtension`/`newMimeType`/`regenerateThumbnails` through to `replaceInPlace` when needed, then also restores metadata fields via `update-meta`. If replace-in-place is unavailable (or throws `CapabilityUnavailableError` and `strict` is false), it falls back to `resolver.resolve('upload')` and uploads the original bytes as a **new** attachment — reported as `'partial'`, not `'restored'`.
4. On a fully in-place `'restored'` outcome, `undo` calls `store.markRestored(snap.id)` and `db.markProcessingReverted(site.name, snap.wpId, snap.operation)` — the latter sets `reverted_at` on the most recent `processing_history` row for that attachment+operation so idempotency checks and cumulative `stats` no longer count it as processed.
5. On a `'partial'` outcome, the snapshot is deliberately left un-restored (not consumed) since the original attachment is still unchanged and still what references point at; the user is told to retry after configuring a replace-in-place backend, or to rewrite references to the new ID.

## Key design decisions

- **SQLite as source of truth, schema v4 migration** (per CLAUDE.md's locked architecture table). `sessions` and `snapshots` were added as migration version 4 in `src/engine/state/schema.ts`'s `MIGRATIONS` array — additive, `IF NOT EXISTS`-safe, applied automatically by the existing migration runner. `snapshots.session_id` has an `ON DELETE CASCADE` FK to `sessions.id`, and both tables cascade from `sites.name`, so the store never needs manual cross-table cleanup beyond dropping the parent row. A companion migration (v5) added `processing_history.reverted_at` specifically to let undo "un-consume" the idempotency/stats trail without deleting history.
- **"Always undoable" as a cross-cutting product principle.** Every command file listed in Requirement 1 independently calls the same three `engine/history` helpers around its mutation loop — there is no central dispatcher forcing this; it's a convention every destructive command follows, the same way `resolveDryRun` is a shared convention for dry-run handling (see CLAUDE.md). `openHistorySession`/`captureSnapshot`/`closeHistorySession` exist specifically to keep each command's integration to ~3 call sites.
- **Binary blobs vs. metadata-only is decided purely by whether `sourceBytes` is passed to `capture()`.** There's no separate flag — commands that mutate file bytes (optimize/convert/resize/remove-bg/replace-in-place push) pass the downloaded original; commands that only touch WP fields (caption/title/describe/tag/rename/metadata) pass `null`. This keeps the store's capture API uniform across very different command shapes.
- **Snapshot capture never blocks the user's operation.** `captureSnapshot()` (in `index.ts`) wraps `store.capture()` in a try/catch and returns `null` on any failure rather than throwing — a full disk or permissions error on the snapshots directory degrades to "this run isn't undoable," not "this run fails." This is a deliberate trade-off: safety-net writes are best-effort, not a hard dependency of the primary operation.
- **Durable-write-before-DB-pointer ordering.** `SnapshotStore.capture()` writes the blob file synchronously (`writeFileSync`) and only then `UPDATE`s the row's `blob_path`/`blob_size` — this was a fixed regression (see CHANGELOG "snapshots are written synchronously and their blobs are integrity-checked... before restore", #92) against a prior fire-and-forget write that could leave a DB row pointing at a not-yet-flushed file.
- **Restore-time integrity verification, not capture-time trust.** `readBlob()` re-verifies file existence, exact byte length, and (when available) SHA-256 hash at restore time rather than assuming a snapshot row implies a good blob — protects against later corruption/truncation of the blob store between capture and restore.
- **Bulk-safe, single-item-immediate — the same pattern as `optimize`/`delete`/etc.** Per CLAUDE.md's "Bulk safety" row: explicit IDs (`--snapshot`, `--attachment`) execute immediately; anything session/bulk-shaped dry-runs unless `--apply`. `undo` implements this directly rather than through the shared `resolveDryRun` helper used by other commands — worth noting as a slight inconsistency (see Error handling below).
- **In-session-only hash dedupe.** Recapturing the same attachment with the same `beforeHash` inside one session returns the existing snapshot instead of writing a duplicate blob. Cross-session dedupe is explicitly not implemented ("kept simple — it's a rare case and complicates restore", per `store.ts` header comment).

## Error handling / edge cases actually implemented

- **REST-only restore is "partial," not "restored."** When no replace-in-place-capable adapter is available, `restoreSnapshot` uploads the original bytes as a new attachment and returns `{ status: 'partial', newAttachmentId }`. The caller (`undo.ts`) explicitly does **not** call `store.markRestored()` for partial outcomes, leaving the snapshot available for a real in-place retry later, and tells the user the new attachment ID plus a suggested `localpress references <old> --update-to <new>` follow-up. (This was itself a bug fix — CHANGELOG 2.2.0: "treat REST-only upload-as-new restore as partial, not restored", #183 — worth calling out since it's the most subtle behavior in the whole subsystem.)
- **Format-changed restore.** If the attachment's current MIME type (fetched live via `getMedia`) differs from the snapshot's recorded `beforeMeta.mimeType` (e.g. `optimize --to webp` ran since the snapshot), `restoreSnapshot` derives the original extension from the snapshot's filename and passes `newExtension`/`newMimeType`/`regenerateThumbnails: true` to `replaceInPlace`. If the resulting URL differs from the current one, it warns the user that content may still reference the stale (post-optimize) URL and suggests `localpress references <id> --scope full`.
- **Missing/truncated/corrupted blob.** `readBlob()` throws distinct, specific errors for a missing file, a byte-length mismatch, and a SHA-256 mismatch — `undo` catches these per-snapshot (not aborting the whole batch) and reports that snapshot as `'failed'` with the error message, continuing to the next.
- **`--strict` mode.** If replace-in-place throws `CapabilityUnavailableError` and `strict` is true, the error propagates instead of falling back to upload-as-new — lets a caller (e.g. an agent that specifically needs an in-place guarantee) opt out of the silent-fallback behavior.
- **Interrupted sessions are still undoable.** `getLastSession()` selects the newest session for the site with `EXISTS (SELECT 1 FROM snapshots WHERE session_id = s.id AND restored_at IS NULL)` rather than requiring `finished_at IS NOT NULL` — a bulk command killed mid-run (Ctrl-C, crash) before `closeSession()` ran still has capturable, restorable snapshots and its session is still what plain `undo` targets (CHANGELOG #108).
- **Empty sessions are pruned automatically.** If every item in a run was idempotency-skipped, `closeSession()` deletes the now-empty session row rather than leaving zero-snapshot noise in `history list`.
- **Rename slug restore.** `SnapshotMeta.slug` is optional and only populated by operations that touch the WP slug (currently `rename`). `restoreSnapshot`'s metadata-only branch spreads `slug` into the `updateMetadata` call only `if (snap.beforeMeta.slug !== undefined)`, so snapshots from operations that never touched the slug (caption, etc.) don't accidentally clear it on restore.
- **`history clear` requires explicit confirmation.** Without `--yes` (or global `--yes`), it warns with the exact count/bytes that would be deleted and exits with code 2 rather than proceeding — this is a manual confirmation gate, not the shared `resolveDryRun` helper, since `history clear` isn't itself dry-run-able (there's no "preview the delete" mode beyond this warning).
- **Retention: overlapping clauses are OR'd, not AND'd.** `store.prune()`'s doc comment is explicit: "a snapshot is dropped if it falls outside ANY clause" — e.g. an `--older-than` clause can drop snapshots even if the `maxSize` cap is already satisfied.
- **Snapshot deletion is best-effort on the filesystem side.** `deleteSnapshot()` wraps blob/directory removal in try/catch and ignores failures — the SQLite row is always removed even if the on-disk file is somehow already gone or unremovable, favoring index consistency over blob cleanup guarantees.
- **Ambiguity flagged:** `markProcessingReverted`'s doc comment in `db.ts` states it "assumes the most recent row for this wp_id+operation is the one being undone... but could mis-attribute if another operation ran on the same attachment between the original run and the undo." This is a known, documented limitation rather than a fully solved edge case — noting it here rather than glossing over it.
- **Interactive/continuous commands are intentionally not snapshotted.** Per CHANGELOG v1.15.0 notes, `edit` and `watch` don't participate in the capture path — `edit`'s round-trip already has its own explicit revert, and `watch` continuously syncs new files, where per-change snapshotting was judged wasteful. This wasn't directly re-verified against `edit.ts`/`watch.ts` source in this pass; flagging it as changelog-sourced rather than freshly confirmed from those two files.

## Testing approach

- **`test/unit/history.test.ts`** — `SnapshotStore` unit tests against a real temp-file SQLite DB (for FK enforcement) and a temp blob directory: session open/close lifecycle, binary vs. metadata-only capture, synchronous blob durability, empty-session pruning, size-cap and age-cap `prune()`, `clear()`, `getLastSnapshotForAttachment` (including the "mark restored → falls back to previous" case), in-session hash dedupe, stats aggregation, slug round-trip in metadata-only snapshots, `getLastSession` finding an interrupted (never-closed) session and skipping a fully-restored one, and all three `readBlob()` integrity failure modes (missing, truncated, hash mismatch) plus the success path with no hash recorded.
- **`test/unit/undo.test.ts`** — `restoreSnapshot()` unit tests using fake `WpBackend` adapters via the `ResolverLike` structural type (no network/WordPress/SSH): format-change restore passes `newExtension`/`newMimeType`/`regenerateThumbnails` and warns about stale URLs; same-format restore omits those options and doesn't warn; in-place restore reports `'restored'`; REST-only fallback reports `'partial'` with the new attachment ID (explicitly tagged as the regression test for issue #119).
- **`test/unit/undo-rename-slug.test.ts`** — targeted regression coverage for the rename-undo slug bug: a metadata-only snapshot carrying a captured `slug` forwards it to `updateMetadata`; a snapshot that never captured a slug leaves it `undefined` rather than clearing it.
- **`test/integration/wp-rest.test.ts`** — against a real Dockerized WordPress instance, `'rename then undo restores the original slug (rename/undo round trip)'` exercises the underlying REST calls directly (not the CLI commands): it renames an attachment's slug via `adapter.updateMetadata({ slug: ... })`, verifies the change, then calls `updateMetadata` again with the original slug to simulate what `undo`'s restore path does, and asserts the slug round-trips. It does not invoke `SnapshotStore` or `restoreSnapshot()` itself — it validates that the REST primitive `undo` depends on behaves correctly against live WordPress, complementing the unit-level `restoreSnapshot()` coverage above.
- No dedicated test file exercises `history.ts`'s or `undo.ts`'s CLI-argument parsing / output formatting directly (as opposed to the underlying `SnapshotStore`/`restoreSnapshot` logic) — coverage is concentrated on the engine layer and the restore function, not on Commander wiring or TUI rendering (`HistoryBrowser.tsx` has no test file found in `test/unit/`).
